## Part 2 · 三个现有功能第一性原理批判

### 2.1 功能一：Conjoint 租房偏好测试

#### 真问题 vs. 假问题

**第一性原理问题**：用户在租房前最焦虑的不是"我不知道自己重视什么"——而是**"我不知道某套房是否踩坑"**。

Conjoint 测试解决的是"自我认知"问题，而北京租房决策的核心矛盾是**信息不对称**，不是偏好不清晰。大多数人在没租过房之前就知道自己"要电梯、不要太吵、通勤不超过45分钟"——问题在于：**这些条件在看房时无从验证**。

#### 方法论硬伤

1. **非补偿性决策被 CBC 完全忽略**。北京租房中存在大量 hard constraint：
   - "价格超过 8000 就不看"（价格阈值）
   - "没有集中供暖就跑"（取暖不可替代）
   - "商水商电直接 pass"（隐性成本不可接受）
   - CBC 假设用户会在不同属性间做 trade-off，但这些 hard constraint 根本不参与权衡

2. **属性设定失真**。CBC 题目呈现的是"电梯/独卫/装修/通勤时间"——但真正踩坑的维度是那些**看不见的**：甲醛浓度、隔音等级、邻居素质、楼层噪音、供暖方式。这些维度在选项中无法量化描述，做题时用户根本想不到。

3. **WTP 数值的可信度存疑**。"愿意为电梯多付 ¥X" 这个结论建立在：用户对选项的理解一致、且没有展示效应影响。但实际上用户在看到"有电梯 vs. 无电梯"时的心理权重，跟真正签合同时完全不同（真实决策时还有预算压力、时间压力、信息不完整等因素）。

#### 结果的可用性问题

测出了 importance% 和 WTP 之后，用户拿这个能做什么？

- 如果是为了**筛房源**：现有产品没有实际房源库，偏好结论没有 actionable 的下游
- 如果是为了**自我认知**：用户付出 10 道题的成本，换到的是一个"你重视通勤（58%）"的结论——大部分用户其实早就知道这个
- 小红书面试官关心的是：**测完偏好→ 然后呢？** 如果 part-worth 不能驱动后续的房源推荐或避坑建议，整个测试就是一个好玩的小游戏，而不是决策工具

#### 用户真实决策路径对比

真实用户（来自牛客/掘金/小宇宙）的决策路径：

```
1. 设定预算上限（hard constraint）
2. 圈定通勤距离（地铁几站/多少分钟，hard constraint）
3. 在平台（贝壳/链家/豆瓣）按价格+区域粗筛
4. 看照片，主观判断装修/采光
5. 约看房，现场感受（但往往发现只有10分钟看）
6. 凭直觉+运气决策
```

没有一个用户说"我做了一个偏好测试来帮我决策"。

### 2.2 功能二：通勤地图（等时圈）

#### 重复工具问题

这个功能确实是**同质化最严重**的一个：

- **高德地图**："从A到B的等时圈"功能早已内置
- **相寓**：2024年2月已上线"双人通勤租房"功能，逻辑完全一致——输入两人通勤地点+时长→生成满足条件的房源列表（[中国新闻网报道](https://www.chinanews.com.cn/cj/2024/02-27/10170610.shtml)）
- **百度/滴答出行**：都有等时圈
- **aqitocigarette.world（深圳版）**：同类思路

所以我们的差异化在哪里？如果只是"也能画等时圈"，这个功能完全可以砍掉。

#### 通勤痛苦的真实来源

根据大量用户声音（[少数派](https://sspai.com/post/103475)、[新浪财经](https://news.sina.cn/2022-09-17/detail-imqqsmrn9473746.d.html)、[澎湃新闻](https://www.thepaper.cn/newsDetail_forward_22457133)），通勤痛苦的本质**不是地铁总分钟数**，而是：

```
真实通勤体验 = 
  步行到站时间（10–20分钟常被忽略）
  + 等车时间（高峰期等3–4趟 = 额外15–20分钟）
  + 换乘次数 × 换乘拥挤系数（每次换乘+10–15分钟有效压力）
  + 被挤压的心理损耗（无法坐下、无法看手机 = 精力消耗）
  + 出站后步行/骑行时间（最后一公里经常被忽略）
```

我们目前用"直线距离/35km/h"估算——这跟真实偏差极大。

**具体案例**：少数派作者从昌平沙河到东城，地图显示约55分钟，实际某天因高峰期排队进站花了2小时、等了3趟才上车，**单程4.5小时**。这种情况下，等时圈"55分钟"毫无参考价值。

#### 双人模式的真实复杂性

现有产品假设：双人通勤 = 两人都满意通勤时间 → 找交集区域

但真实情侣/室友的决策要复杂得多（来自自如整租攻略 [2017](https://jiaju.sina.cn/zixun/20170713/6291187295446172626.shtml) 及多个用户反馈）：

- **权重不对等**："我让步给他更近，但他要在租金上让步"——本质是一个多目标优化，而不是简单取交集
- **时间偏好不同**：一个人弹性上班，另一个人9点打卡——同样50分钟通勤，体验完全不同
- **方向偏好**：坐地铁方向不同，"顺行"和"逆行"的拥挤程度差3–4倍

#### 找站之后呢？

用户确定了目标地铁站后，下一步是什么？产品止步于"找到站"，但用户接下来需要：
1. 看这个站附近有哪些房源（接房源）
2. 了解这个区域的小区评价（接小区点评）
3. 知道这个区域的租金均价（接预算规划）

等时圈孤立存在，没有与其他功能形成闭环。

### 2.3 功能三：小区点评 + 匹配度

#### 数据真实性是核心护城河

如果数据是 mock 的，整个功能就是一个**UI展示**，没有产品价值。真正的护城河在于：

- **小红书 UGC**：有真实内容，但非结构化、无法系统汇总
- **链家/贝壳点评**：有，但维度极少（总分+文字评价），且存在刷分嫌疑
- **差异化机会**：能否聚合小红书/知乎/豆瓣/微博上关于某小区的所有真实声音，AI 结构化处理成评分？这才是真正的护城河

#### 评分维度是否覆盖真实痛点？

现有维度：噪音、隔音、物业、安全、生活配套、性价比

这些确实是 Maslow 基础层。但根据调研，**频率最高但评分体系没有的维度**包括：

| 缺失维度 | 真实用户频率 | 为什么重要 |
|---------|------------|----------|
| 供暖方式（集中/自采暖） | 极高 | 北方冬天决定居住质量，且是硬伤无法改变 |
| 商水商电 vs. 民水民电 | 极高 | 月费用差距可达 300–800 元，是隐藏成本 |
| 甲醛风险 | 高 | 新装修小区系统性风险，用户普遍焦虑但无从查 |
| 楼层/采光时长 | 高 | 1–2楼蟑螂+阴暗是共识，但评分里没有楼层信息 |
| 外卖/便利店密度 | 中 | 年轻人高度依赖，"最近美团骑手不接单"是真实抱怨 |
| 电梯老旧/等待时间 | 中 | 高层公寓电梯是日常体验核心 |
| 邻居素质（含极端邻居风险） | 中 | 回迁房 = 高风险标签，用户需要这个信号 |
| 女生安全感（夜间路灯/监控） | 中 | 女生独居群体的核心决策因子 |

#### 匹配度算法的根本性错位

当前逻辑：**Conjoint 偏好（房间维度）× 小区评分（小区维度）**

这两个维度**根本就不在同一层次**：

- Conjoint 测的是：电梯、独卫、装修风格、面积——这是**房间/户型**维度
- 小区点评测的是：物业、噪音、生活配套——这是**小区/社区**维度

用"我重视装修（房间层）"去匹配"物业好（小区层）"，逻辑上是断裂的。

**修复方案**：
1. 把 Conjoint 拆成两层：房间偏好（装修/面积/独卫）+ 小区偏好（噪音/安全/配套）
2. 或者直接在小区点评功能中增加"根据用户关键词偏好权重重算匹配分"（比如用户最在意噪音，就给噪音分加权3倍）

## Part 3 · 模拟用户访谈（半结构化）

### 角色 A：刚毕业北漂工程师

> **画像**：李明，25岁，后端开发，月薪18k，到手约13k，租房预算 8k 以内，目标区域国贸/CBD/三里屯周边，单身独居，第一次在北京租房

## Part 4 · 综合建议

### A. 三个现有功能的优化方向（按 ROI 排序）

#### 功能一：Conjoint 偏好测试

| 优先级 | 具体动作 | 预期效果 |
|--------|---------|---------|
| 🔴 高 ROI | **增加 Hard Constraint 层**：在测试开始前，让用户先勾选"必须满足"条件（民水民电 / 集中供暖 / 独卫 / 价格上限 / 无隔断房），这些不进入 CBC 权衡，直接作为过滤器 | 解决 CBC 方法论最大硬伤；让结果更接近真实决策逻辑 |
| 🔴 高 ROI | **偏好→避坑清单打通**：测试结果不仅输出 importance%，还自动生成"根据你的偏好，看房时必须检查这10件事"。例如：重视安静→看房必听：打开水龙头听管道声音；重视采光→晴天下午2点去看房 | 让测试有 actionable 下游；增加实用价值 |
| 🟠 中 ROI | **去掉通勤作为 CBC 属性，改为独立的硬性筛选**：通勤时间是强 hard constraint（"不超过 X 分钟"），不适合参与属性间权衡，单独成一个滑块设置即可 | 减少 CBC 题目干扰；与通勤地图功能对接 |

#### 功能二：通勤地图

| 优先级 | 具体动作 | 预期效果 |
|--------|---------|---------|
| 🔴 高 ROI | **引入换乘惩罚系数和拥挤线路标记**：在等时圈计算中，对需要换乘2次以上的路线增加20–30%时间惩罚系数；标记早高峰人满为患的线路（6号线、4号线、10号线等）用红色警示 | 让通勤预测更接近真实体验；这是现有所有竞品都没有做的差异点 |
| 🔴 高 ROI | **等时圈→小区列表闭环**：用户点击地图上一个站后，直接展示该站步行 15 分钟内的小区列表，并与小区点评数据联动 | 打通通勤地图和小区点评，形成完整决策链 |
| 🟠 中 ROI | **双人模式升级为"通勤权衡计算器"**：增加"我愿意多通勤X分钟 ↔ TA需要少付¥Y/月"的权衡视图，让情侣/室友显性化地做 trade-off，而不是简单取时间交集 | 解决情侣找房的真实决策复杂性；差异化很强 |

#### 功能三：小区点评

| 优先级 | 具体动作 | 预期效果 |
|--------|---------|---------|
| 🔴 高 ROI | **增加"隐藏硬伤"维度**：供暖方式（集中/自采暖）、水电类型（民/商）、建筑类型（回迁房高风险标记）——这三个维度是最多人踩坑但评分系统没有的，且是 hard constraint 层 | 切中真实痛点#6（商水商电陷阱），差异化明显 |
| 🔴 高 ROI | **修复匹配度算法的逻辑错位**：将偏好拆为"房间层偏好"（装修/面积/独卫）和"小区层偏好"（噪音/安全/配套），匹配度只用后者与小区评分对接 | 让匹配度结论有逻辑一致性；提升可信度 |
| 🟠 中 ROI | **增加"女生安全专项评分"**：从评论中提取女生独居相关信号（门禁严格度、夜间路灯、邻居素质），生成专项分，并支持"仅看女生用户评价"过滤 | 切入女生独居用户群体；这是强传播触发点 |

### B. 是否要砍掉/合并某个功能？

**结论：通勤地图不该砍，但必须重新定位；Conjoint 测试需要降权重。**

**通勤地图**：目前是三个功能里**用户感知价值最直观**的——画面有交互、有地图、有视觉反馈。问题不是"这个功能没价值"，而是"功能做完就断了"。优化方向是：让它成为入口，而不是终点。

**Conjoint 测试**：目前是**展示技术亮点最强**的功能（面试 Demo 场景），但**用户实用价值最低**。建议：保留作为 onboarding 收集用户偏好的手段，但大幅简化（从10道题减到5道，去掉对用户价值最低的维度），并把结果直接接到"避坑清单生成"和"小区匹配"上。

**小区点评**：三个功能里**护城河潜力最大**——只要数据是真实的，这个功能能做出别人做不出来的东西。问题是 Demo 用 mock 数据完全体现不出价值。建议：在 Demo 中至少为3–5个真实北京小区抓取并结构化真实评论，哪怕只有10条原始贴，也比100条 mock 数据说服力强10倍。

**功能间是否有重叠**：目前三个功能几乎完全独立，没有闭环。最大的设计缺陷不是重叠，而是**断链**——通勤地图找到站之后没有下游，Conjoint 测完偏好之后没有下游，小区点评匹配度的算法来源没有正确上游。

---

### C. 是否要新增一个核心功能？

**强烈建议新增：「看房避坑清单生成器」**

**理由**：

1. **切中痛点 #1 和 #11**：信息不对称 + 毕业生不知道该问什么，是用户频率最高、严重度最高的痛点
2. **AI Demo 展示性极强**：输入"我要看的小区/房型/预算"→ AI 根据偏好个性化生成"看房必问30题"，这个交互极适合展示 AI 能力
3. **零竞争对手**：高德/贝壳/链家没有一家做这个，这是空白市场
4. **强社交传播触发**：用户生成后天然想截图分享（"你帮我生成了一份看房检查清单！"）

**功能设计建议**：

```
用户输入：
  - 我看的是：[合租/整租/公寓]
  - 我最在意：[通勤/安静/安全/价格]
  - 我是第一次租房吗？[是/否]

AI 输出：
  必问问题（3类）：
    📋 证件核查（3题）：是否有房产证？供暖方式？水电类型？
    🔍 现场检查（10题）：打开水龙头听管道；关灯打开闪光灯看霉斑；...
    🤝 合同条款（5题）：甲醛问题退租条款？设施维修责任归属？...
  
  根据用户偏好定制的额外问题：
    [若在意噪音] → 问：楼上楼下入住率？隔壁是否有小孩？
    [若女生独居] → 问：电梯有无监控？门禁是否刷卡？
    [若合租]     → 问：室友是否已入住？几人共用一个卫生间？
```

**次选方向：「房源真实性交叉验证」**

用户贴入链家/自如链接 → AI 拉出小区历史舆情（小红书/知乎/豆瓣关键词聚合）+ 与当前标价对比是否高于周边均价 + 标注高风险信号（新装修+无甲醛承诺、回迁房标签等）。

这个功能的技术难度更高，但面试 Demo 中展示"AI 读一个链接然后输出一份风险报告"，视觉冲击感极强。

### D. 整体性建议：产品定位与三功能闭环

#### 一句话定位

> **租房雷达 = 北京年轻人的「租前决策副驾驶」——不帮你找房，帮你在找房全程少踩坑、做对决策。**

区分于贝壳/链家（房源平台）、高德（导航工具）、自如（公寓运营商）。

#### 三功能闭环设计

```
                    ┌──────────────────────┐
                    │   偏好收集 + 避坑清单  │
                    │   (Conjoint 简化版)    │
                    │   ↓                   │
                    │   我要：独卫+安静+8k   │
                    │   我怕：甲醛/噪音/坑   │
                    └──────────┬───────────┘
                               │ 偏好传递
                 ┌─────────────▼──────────────┐
                 │      通勤地图               │
                 │   输入公司 → 画等时圈        │
                 │   → 推荐候选地铁站          │
                 │   → 点击站 → 进入小区列表   │
                 └─────────────┬──────────────┘
                               │ 区域传递
                 ┌─────────────▼──────────────┐
                 │      小区点评 + 匹配度        │
                 │   真实评分（噪音/安全/硬伤）  │
                 │   ×                         │
                 │   你的偏好权重               │
                 │   =                         │
                 │   「为你量身定制的匹配分」    │
                 │   + 生成看房避坑清单         │
                 └────────────────────────────┘
```

**关键设计原则**：
- 偏好测试是**入口**，不是核心功能
- 通勤地图是**筛选器**，不是终点
- 小区点评是**决策层**，是真正拉开差距的护城河

面试官问"你的产品是什么"，答案是：**"一个帮你在租房决策全链路少踩坑的 AI 副驾驶。从知道自己要什么、到找对区域、到看对房子——三步形成闭环。"**

---

## TL;DR · 3 个最关键的产品决策建议

**1. 砍 Conjoint 的"心理测试"包装，改造成"偏好收集 + 避坑清单生成"的功能入口。** CBC 的 part-worth 对用户没有直接可用价值；真正有价值的是：用偏好驱动个性化的看房检查清单，这才是「AI Demo 能力」的最佳展示场景。

**2. 通勤地图的差异化不在于"能画等时圈"，而在于"换乘惩罚 + 真实拥挤感知 + 站→小区闭环"。** 相寓已于 2024 年 2 月上线双人通勤功能；如果不做差异化，这个功能毫无竞争壁垒，建议增加换乘系数校正和下游房源/小区联动，才能从"又一个等时圈工具"变成"真正有用的北京租房通勤分析"。

**3. 小区点评的核心价值在于数据的真实性，而非评分维度的完整性。** Demo 中哪怕只有 3 个真实北京小区的真实聚合评论（从小红书/豆瓣提取），说服力也远超 100 条 mock 数据；同时必须增加"供暖方式/商水商电/回迁房标识"三个「隐形硬伤」维度，这是与现有所有租房平台最明确的差异化方向。
